## 代理分工

- 主代理负责理解目标、拆分任务、定义验收标准、派单、汇总结果和最终判断。
- 用户指定子代理执行的任务，主代理不得自行接手其实现、测试或 review，除非先向用户说明原因并获得确认。
- 禁止使用 GPT 子代理；需要派单时只使用 OSS 子代理。
- `oss_flash` / `oss_pro` 用于文本任务；`oss_mimo` / `oss_minimax` 用于多模态或视觉任务。

## 子代理规则

- 派单必须边界明确、可验收；结果只需返回结论、关键发现、变更文件、测试结果和风险。
- 同一问题或代码区域优先续问已有子代理；目标、范围或验收标准明显变化时，新开干净子代理。
- 复用子代理前必须重申当前目标和验收标准；长时间运行或多轮改动后的子代理只作为上下文顾问，关键判断用干净子代理复核。
- 子代理之间不直接通信，所有发现由主代理汇总。

## 异常与日志

- 子代理初始化失败、超时、挂起、结果不完整或疑似偏离任务时，先向用户报告状态，不得默认接手。
- 汇总结果时区分：已派出、运行中、已完成、已关闭、失败、主代理补做。
- 避免无边界探索、重复读取、低价值派单和低收益并发。
- 长命令输出写入 `target/codex-logs/*.log`；成功只汇报命令、退出码和结论，失败只读取关键错误、匹配摘要和末尾最多 80 行。
- 默认使用低日志命令，例如 `cargo test -q`、`cargo build -q --release`、`cargo install -q --path . --force`。
